Manage Apps Script Settings and Checkpoints with PropertiesService

Store small configuration values, processing checkpoints, and user-specific preferences without placing them directly in a spreadsheet.

Published: 2026-07-22Last updated: 2026-07-22
In this guide
  1. What belongs in PropertiesService
  2. Choose the correct store
  3. Example
  4. Operating rules

What belongs in PropertiesService

PropertiesService stores string key-value pairs within an Apps Script project. It is useful for endpoints, notification IDs, the last processed row, timestamps, and other small configuration or state values.

It is not an unlimited database for large records or highly sensitive credentials. Review storage limits and access scope before using it.

Choose the correct store

  • Script Properties: settings shared by the whole script.
  • User Properties: preferences specific to the current user.
  • Document Properties: values associated with a document in supported add-on contexts.

Example

function saveSettings() {
  const props = PropertiesService.getScriptProperties();
  props.setProperties({
    SLACK_CHANNEL_ID: 'C0123456789',
    LAST_ROW: '250',
    LAST_RUN_AT: new Date().toISOString()
  });
}

function loadSettings() {
  const props = PropertiesService.getScriptProperties().getProperties();
  const lastRow = Number(props.LAST_ROW || 1);
  return { ...props, lastRow };
}

Operating rules

  1. Use a consistent naming convention for keys.
  2. Serialize numbers or JSON when saving and restore the type when reading.
  3. Fail early with a clear message when required settings are missing.
  4. Delete obsolete keys instead of leaving stale configuration behind.
  5. Treat sensitive values carefully in permissions and logs, and consider a dedicated secret-management service when appropriate.
Key point

Separating configuration from business data makes the script easier to maintain when the spreadsheet layout changes.
