Design Error Logging and Safe Retries in Apps Script

Record failures from APIs, email, and file operations, then retry only transient errors without creating duplicate work.

Published: 2026-07-22Last updated: 2026-07-22
In this guide
  1. Do not hide failures
  2. Errors that may be retried
  3. Exponential-backoff example
  4. Operational design

Do not hide failures

Automations often fail when nobody is watching the screen. Swallowing every exception in a catch block makes it impossible to distinguish completed work from missing work.

Record the operation, record ID, time, message, and retry count. Avoid logging personal data, access tokens, or other secrets.

Errors that may be retried

  • Temporary network failures and timeouts.
  • Rate limits such as HTTP 429.
  • Temporary server errors in the HTTP 500 range.
  • Do not automatically retry invalid input, missing permissions, authentication failures, or nonexistent IDs.

Exponential-backoff example

function withRetry_(task, maxAttempts = 4) {
  let lastError;
  for (let attempt = 1; attempt <= maxAttempts; attempt++) {
    try {
      return task();
    } catch (error) {
      lastError = error;
      console.error(JSON.stringify({ attempt, message: error.message }));
      if (attempt < maxAttempts) {
        Utilities.sleep(Math.pow(2, attempt - 1) * 1000);
      }
    }
  }
  throw lastError;
}

Operational design

  1. Assign a unique ID to each unit of work.
  2. Store completed, failed, and retry-pending states.
  3. Make reruns idempotent so they do not create duplicates.
  4. Notify an administrator only after the final failure.
  5. Review failures regularly and group them by root cause.
Key point

More retries are not a substitute for diagnosis. Structured logs should make failure categories visible.
