Reduce API Calls and Heavy Work with Apps Script CacheService

Temporarily reuse API responses and reference data to improve Apps Script speed without treating the cache as permanent storage.

Published: 2026-07-22Last updated: 2026-07-22
In this guide
  1. What to cache
  2. Cache scopes
  3. API-response example
  4. Design rules

What to cache

CacheService temporarily stores values that can be reused to reduce external requests or expensive calculations. It should not hold the authoritative copy of configuration or irreplaceable records.

Cached entries can disappear before their requested expiration, so the application must be able to rebuild the value from the original source.

Cache scopes

  • Script cache: temporary data shared by script users.
  • User cache: temporary data specific to a user.
  • Document cache: temporary data associated with a document context.

API-response example

function getReferenceData() {
  const cache = CacheService.getScriptCache();
  const key = 'reference-data-v1';
  const cached = cache.get(key);
  if (cached) return JSON.parse(cached);

  const response = UrlFetchApp.fetch('https://api.example.com/reference');
  const data = JSON.parse(response.getContentText());
  cache.put(key, JSON.stringify(data), 900);
  return data;
}

function clearReferenceCache() {
  CacheService.getScriptCache().remove('reference-data-v1');
}

Design rules

  1. Include the data type and version in the key.
  2. Choose an expiration based on how often the source changes.
  3. Remove related entries after an authoritative update.
  4. Store a smaller summary instead of an oversized payload.
  5. Test that a cache miss still produces a correct result.
Key point

Use caching as an optional acceleration layer. Correctness must not depend on the cache being present.
