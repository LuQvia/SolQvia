Handle Google Forms Submit Event Objects Correctly in Apps Script

Understand the different event objects from Forms and response spreadsheets, and build notifications that survive column changes.

Published: 2026-07-22Last updated: 2026-07-22
In this guide
  1. The event depends on the trigger source
  2. Avoid depending on column order
  3. Reliability checks
  4. Troubleshooting

The event depends on the trigger source

A submit trigger attached to the Google Form and one attached to its response spreadsheet provide different event-object structures.

Form triggers expose a FormResponse, while spreadsheet triggers can provide namedValues, values, and range. Confirm where the trigger is installed before adapting sample code.

Avoid depending on column order

function onFormSubmit(e) {
  if (!e || !e.namedValues) {
    throw new Error('This handler requires a spreadsheet form-submit trigger.');
  }

  const get = label => String((e.namedValues[label] || [''])[0]).trim();
  const message = [
    `Requester: ${get('Name')}`,
    `Type: ${get('Request type')}`,
    `Details: ${get('Details')}`
  ].join('\n');

  console.log(message);
}

Reliability checks

  • Use an installable trigger when the workflow requires authorization.
  • Changing a question title also changes the namedValues key, so define a change process.
  • Validate empty values even for required questions.
  • Store a row number or response ID to avoid duplicate notification.
  • Keep test submissions separate from production reporting.

Troubleshooting

  1. Review the trigger execution history.
  2. Log only safe event key names.
  3. Compare the form question titles with the code keys.
  4. For authorization errors, check the trigger owner and consent status.
  5. Test that resubmission or reruns do not duplicate the record.
Key point

Coordinate form-editing rules with the people responsible for maintaining the script.
