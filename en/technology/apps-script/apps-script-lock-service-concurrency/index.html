Control Concurrent Apps Script Runs with LockService

Prevent duplicate sequence numbers and conflicting updates when forms, triggers, or web-app requests run at nearly the same time.

Published: 2026-07-22Last updated: 2026-07-22
In this guide
  1. What concurrent execution can break
  2. Lock scopes
  3. Sequence-number example
  4. Safe patterns

What concurrent execution can break

When multiple form submissions or web-app requests run at nearly the same time, two executions may read the same last row or sequence value and write duplicate results.

Treat the read-modify-write section as one mutually exclusive operation to reduce collisions.

Lock scopes

  • Script lock: permits one protected operation across the script.
  • User lock: prevents overlapping operations for the same user.
  • Document lock: coordinates access within a document context.

Sequence-number example

function issueNumber() {
  const lock = LockService.getScriptLock();
  lock.waitLock(10000);
  try {
    const props = PropertiesService.getScriptProperties();
    const current = Number(props.getProperty('NEXT_NUMBER') || 1);
    props.setProperty('NEXT_NUMBER', String(current + 1));
    return `REQ-${String(current).padStart(6, '0')}`;
  } finally {
    lock.releaseLock();
  }
}

Safe patterns

  • Keep the protected section as short as possible.
  • Use a timeout instead of waiting forever.
  • Release the lock in a finally block.
  • Move slow API calls and heavy calculations outside the lock.
  • Combine locks with unique IDs and duplicate validation.
Key point

A lock reduces race conditions, but business-level uniqueness should also be validated.
