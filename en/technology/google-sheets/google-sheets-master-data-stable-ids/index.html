Use Stable IDs Instead of Names for Spreadsheet Master Data

Prevent incorrect lookups and broken history by assigning immutable IDs to staff, locations, products, and other master records.

Published: 2026-07-22Last updated: 2026-07-22
In this guide
  1. Names are not reliable keys
  2. Core master columns
  3. Lookup example
  4. Operating rules

Names are not reliable keys

People, locations, and products can share a name or be renamed later. Using a display name as the lookup key can match the wrong record or split history.

Assign an immutable staff ID, location code, product code, or other unique identifier.

Core master columns

  • id: unique, stable identifier.
  • display_name: user-facing label.
  • status: active or inactive.
  • valid_from and valid_to when effective dates are needed.
  • attributes such as team, category, rate, or region.

Lookup example

=IFERROR(VLOOKUP($A2,StaffMaster!$A:$F,2,FALSE),"Unknown ID")

Operating rules

  1. Define who issues IDs and how.
  2. Never reuse an ID for a different entity.
  3. Update the display name without changing the ID.
  4. Use a status instead of deleting historical entities.
  5. Show both ID and name in validation choices when useful.
Key point

Separating identity from display text protects historical consistency.
