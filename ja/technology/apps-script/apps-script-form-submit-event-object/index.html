GASのフォーム送信イベントオブジェクトを正しく扱う

Googleフォームと回答先スプレッドシートで異なるイベントデータを理解し、列追加にも強い通知処理を作ります。

公開日: 2026-07-22最終更新: 2026-07-22
この記事の内容
  1. トリガーの設置場所でデータが変わる
  2. 列順に依存しない通知
  3. 安定させるポイント
  4. 障害調査の方法

トリガーの設置場所でデータが変わる

Googleフォーム本体に設定した送信トリガーと、回答先スプレッドシートに設定した送信トリガーでは、イベントオブジェクトの構造が異なります。

フォーム側ではFormResponse、スプレッドシート側ではnamedValuesやvalues、rangeなどを扱います。サンプルコードをそのまま移植せず、設置場所を確認します。

列順に依存しない通知

function onFormSubmit(e) {
  if (!e || !e.namedValues) {
    throw new Error('This handler requires a spreadsheet form-submit trigger.');
  }

  const get = label => String((e.namedValues[label] || [''])[0]).trim();
  const message = [
    `申請者: ${get('氏名')}`,
    `種類: ${get('申請種別')}`,
    `内容: ${get('内容')}`
  ].join('\n');

  console.log(message);
}

安定させるポイント

  • 単純トリガーではなく、必要な権限を持つインストール型トリガーを設定する。
  • 質問タイトルを変更するとnamedValuesのキーも変わるため、変更手順を決める。
  • 必須回答でも空文字を想定して検証する。
  • 送信行番号や回答IDを保存し、二重通知を防ぐ。
  • テスト送信のデータを本番集計から分離する。

障害調査の方法

  1. トリガーの実行履歴を確認します。
  2. イベントに存在するキー名だけを安全にログへ出します。
  3. フォーム質問名とコード上のキーを比較します。
  4. 権限エラーの場合はトリガー作成者と認可状態を確認します。
  5. 再送時に同じ回答が重複処理されないかテストします。
ポイント

フォームを編集する担当者とスクリプトを保守する担当者の変更ルールを合わせておくことが重要です。
