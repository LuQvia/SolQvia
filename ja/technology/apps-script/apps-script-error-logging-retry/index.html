Google Apps Script

GASのエラーログと再試行を設計する方法

外部API、メール、ファイル処理の失敗を記録し、一時的な障害だけを安全に再試行する実務設計を解説します。

公開日: 2026-07-22最終更新: 2026-07-22
この記事の内容
  1. エラーを隠さない
  2. 再試行してよい失敗
  3. 指数バックオフの例
  4. 運用設計

エラーを隠さない

自動化では、利用者が画面を見ていない時間に失敗することがあります。catchで例外を握りつぶすと、処理済みと未処理の判別ができなくなります。

最低限、処理名、対象ID、発生時刻、エラーメッセージ、再試行回数を記録します。個人情報やアクセストークンはログへ出さないようにします。

再試行してよい失敗

  • 一時的な通信エラーやタイムアウト。
  • HTTP 429などのレート制限。
  • HTTP 500番台の一時的なサーバー障害。
  • 再試行してはいけない例: 認証失敗、入力形式不正、権限不足、存在しないID。

指数バックオフの例

function withRetry_(task, maxAttempts = 4) {
  let lastError;
  for (let attempt = 1; attempt <= maxAttempts; attempt++) {
    try {
      return task();
    } catch (error) {
      lastError = error;
      console.error(JSON.stringify({ attempt, message: error.message }));
      if (attempt < maxAttempts) {
        Utilities.sleep(Math.pow(2, attempt - 1) * 1000);
      }
    }
  }
  throw lastError;
}

運用設計

  1. 処理対象に一意IDを付けます。
  2. 成功、失敗、再試行待ちの状態を保存します。
  3. 同じ対象を再実行しても二重登録しない構造にします。
  4. 最終失敗時だけ管理者へ通知します。
  5. 定期的に失敗一覧を確認し、原因別に集計します。
ポイント

再試行回数を増やすだけでは根本解決になりません。失敗理由を分類できるログが重要です。
